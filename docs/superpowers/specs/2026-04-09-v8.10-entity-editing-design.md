# V8.10 ŌĆö Entity Editing Foundation + Core Entities

**Date:** 2026-04-09
**Branch:** `feature/v8.10-entity-editing` (to be created)

## Problem

Users can create 16 entity types but can only edit 3 (cooperative profile, legal document, alignment interests). This forces users to delete and recreate entities to correct mistakes. V8.10 adds edit support for 5 core entities; V8.11 extends to 8 more using the same patterns.

## Scope

### In scope
- Frontend `canEdit` permission helpers (`entity-permissions.ts`)
- Shared form components for Proposal, Agreement, Campaign
- Edit routes (form-page pattern) for Proposal, Agreement, Campaign
- Edit modals (modal-reuse pattern) for Task, Expense
- Backend: widen Campaign `updateCampaign` to allow edits in `active` state
- Tests: API test for campaign active update, E2E for proposal edit, E2E for task edit

### Out of scope
- Entity deletion UI
- Auto-save / draft management
- V8.11 entities (officer, compliance, notice, fiscal period, commerce, alignment)

## Design

### Two UI Patterns

**Form-page reuse** (Proposal, Agreement, Campaign): Extract form fields from `new/+page.svelte` into a shared `<EntityForm>` component. Create `[id]/edit/+page.svelte` that loads the entity and renders the same form pre-filled. Detail pages get an "Edit" link in the action bar when `canEdit` returns true.

**Modal reuse** (Task, Expense): The existing create modal gains an `editingEntity` state variable. When set, the modal pre-fills fields and submits to a `?/update` action instead of `?/create`. Edit buttons appear on list/board items when `canEdit` returns true.

### Form Component Contract

Form components:
- Accept `initialValues?: Partial<EntityFields>` for pre-filling
- Accept `disabledFields?: string[]` for locking fields (campaign active state)
- Render labeled field `<div>` blocks with `name` attributes
- Do NOT render `<form>`, submit button, or cancel link
- Parent page wraps in `<form method="POST" action="?/..." use:enhance>`

### Permission Helpers

`apps/web/src/lib/utils/entity-permissions.ts` ŌĆö pure functions, one per entity:

| Entity | Rule |
|--------|------|
| Proposal | `status === 'draft' && authorDid === userDid` |
| Agreement | `status === 'draft' && authorDid === userDid` |
| Campaign | `status === 'draft' \|\| status === 'active'` |
| Task | `status !== 'done' && status !== 'cancelled'` |
| Expense | `(status === 'submitted' \|\| status === 'draft') && memberDid === userDid` |

Frontend-only. Backend already enforces equivalent checks in service methods.

### Backend Changes

1. `funding-service.ts:202` ŌĆö widen status check: `campaign.status !== 'draft'` ŌåÆ `campaign.status !== 'draft' && campaign.status !== 'active'`
2. `funding-service.ts` ŌĆö add guard rejecting `fundingModel` changes when `campaign.status === 'active'` (defense-in-depth; frontend also locks this field)

### Proposal Field Name Translation

The Proposal type and `updateProposal` API use `votingMethod`/`quorumRequired`/`votingEndsAt`, but the create API uses `votingType`/`quorumType`/`closesAt`. ProposalForm uses create-API names. The edit page.server.ts translates in both directions (load ŌåÆ initialValues, FormData ŌåÆ update API call).

### Edit Page Editability Guards

Every edit `+page.server.ts` load function checks editability (status + ownership). If not editable, redirects to the detail page. Handles bookmarked/shared edit URLs and entity state changes.

### Campaign Active-State Field Locking

When editing an active campaign, structural fields are locked:
- Locked: `tier`, `campaignType`, `fundingModel`
- Editable: `title`, `description`, `goalAmount`, `goalCurrency`, `endDate`

Frontend: `disabledFields` prop on `CampaignForm` + `disabled` attribute + hidden `<input>` for disabled fields.
Backend: guard rejects `fundingModel` changes for active campaigns.

### Campaign Cents/Dollars

Create page converts `dollars * 100` before API call. Edit page converts `cents / 100` for display and `dollars * 100` on submit.

### Modal Re-render Strategy

Svelte 5 inputs don't re-render values when props change. Use `{#key editingEntity?.id ?? 'create'}` blocks around modal form contents to force DOM teardown/recreation when switching entities.
