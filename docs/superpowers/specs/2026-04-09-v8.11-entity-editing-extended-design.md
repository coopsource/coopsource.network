# V8.11 — Entity Editing Extended

**Date:** 2026-04-09
**Branch:** `feature/v8.11-entity-editing-extended` (to be created)

## Problem

V8.10 added edit UI for 5 entities using two patterns (form-page reuse and modal reuse). 8 entities remain without edit UI. V8.11 applies the modal-reuse pattern to 6 of those. Two are skipped: Alignment Outcomes (immutable by design) and Member Notices (fire-and-forget, no draft state).

## Scope

### In scope (6 entities)

| Entity | UI Pattern | Backend Work |
|--------|-----------|-------------|
| Officer | Modal reuse (admin tab) | New PUT endpoint + service + client + validation schema |
| Compliance Item | Modal reuse (admin tab) | New PUT endpoint + service + client + validation schema |
| Fiscal Period | Modal reuse (admin tab) | New PUT endpoint + service + client + validation schema |
| Commerce Listing | Modal reuse (listings page) | Backend status guard (PUT exists) |
| Commerce Need | Modal reuse (needs page) | Missing client method + backend status guard (PUT exists) |
| Member Roles | Inline dropdown (members table) | None — PUT + client exist |

### Prerequisite: Admin page extraction

The admin page is a 553-line monolith hosting 5 tabs and 7 modals. Before adding edit modals, extract each tab into its own component: `OfficersTab.svelte`, `ComplianceTab.svelte`, `FiscalPeriodsTab.svelte`. The main admin page becomes a thin tab router.

### Out of scope

- Alignment Outcomes — immutable by design (only status transitions exist)
- Member Notices — fire-and-forget (no draft state; `create()` immediately sets `sent_at`)
- Entity deletion UI (separate concern)

## Design

### canEdit Rules

Added to `apps/web/src/lib/utils/entity-permissions.ts`:

| Entity | Rule |
|--------|------|
| Officer | `status === 'active'` |
| Compliance Item | `status === 'pending' \|\| status === 'overdue'` |
| Fiscal Period | `status === 'open'` |
| Commerce Listing | `status === 'active' \|\| status === 'paused'` |
| Commerce Need | `status === 'open' \|\| status === 'matched'` |
| Member Roles | Always editable (permission-gated, not status-gated) |

### Backend: 3 New PUT Endpoints

All three follow the same pattern: validate status, parse body, update fields, return updated entity.

**`PUT /api/v1/admin/officers/:id`**
- Updatable fields: `title`, `termEndsAt`, `responsibilities`
- NOT updatable: `officerDid`, `appointmentType`, `appointedAt` (identity/historical facts)
- Status guard: `status === 'active'`
- Permission: `officer.manage`
- Validation: `UpdateOfficerSchema` (Zod `.pick().partial()`)

**`PUT /api/v1/admin/compliance/:id`**
- Updatable fields: `title`, `description`, `dueDate`, `filingType`
- Status guard: `status === 'pending' || status === 'overdue'`
- Permission: `compliance.manage`
- Validation: `UpdateComplianceItemSchema`

**`PUT /api/v1/admin/fiscal-periods/:id`**
- Updatable fields: `label`, `startsAt`, `endsAt`
- Status guard: `status === 'open'`
- Permission: `compliance.manage`
- Validation: `UpdateFiscalPeriodSchema`

### Backend: Commerce Status Guards

Add status checks to existing `updateListing()` and `updateNeed()` service methods:
- Listing: reject updates when `status === 'archived'`
- Need: reject updates when `status === 'fulfilled' || status === 'cancelled'`

### Backend: Missing Client Method

Add `updateCommerceNeed(id, body)` to `apps/web/src/lib/api/client.ts`.

### Frontend: Admin Page Extraction (V8.11a)

Extract from `apps/web/src/routes/(authed)/coop/[handle]/admin/+page.svelte`:

- `apps/web/src/lib/components/admin/OfficersTab.svelte` — officers table + create modal + end-term confirm
- `apps/web/src/lib/components/admin/ComplianceTab.svelte` — compliance table + create modal + complete confirm
- `apps/web/src/lib/components/admin/FiscalPeriodsTab.svelte` — fiscal periods table + create modal + close confirm

Each component receives `data` and `form` as props and emits form submissions via standard `<form method="POST" action="?/...">` + `use:enhance`. The parent page becomes a tab router that renders the active tab component.

NoticesTab and AgentsTab can also be extracted for consistency, but are not required for V8.11 (they don't get edit modals).

### Frontend: Admin Entity Edit Modals (V8.11b)

Each extracted tab component gets the V8.10 modal-reuse pattern:
- `editingOfficer = $state<Officer | null>(null)` (etc.)
- Edit button in table rows, gated by `canEditX(entity)`
- Shared modal switches between create/edit via `{#key}` block
- Form action switches: `?/createOfficer` vs `?/updateOfficer`
- Hidden `id` input for edit mode

Admin `+page.server.ts` gets 3 new form actions: `updateOfficer`, `updateCompliance`, `updateFiscalPeriod`.

### Frontend: Commerce Edit Modals (V8.11c)

Same modal-reuse pattern applied to:
- `apps/web/src/routes/(authed)/coop/[handle]/commerce/listings/+page.svelte`
- `apps/web/src/routes/(authed)/coop/[handle]/commerce/needs/+page.svelte`

Each page gets `editingListing`/`editingNeed` state, edit buttons on cards, shared modal with `{#key}`.

Server files get `updateListing`/`updateNeed` form actions.

### Frontend: Member Role Inline Picker (V8.11d)

In `apps/web/src/routes/(authed)/coop/[handle]/members/+page.svelte`:

- "Edit roles" button appears on each member row (admin-only via permission check)
- Clicking opens an inline multi-select dropdown below the roles badges
- Checkboxes for: `member`, `coordinator`, `admin`, `observer`
- Save/Cancel buttons
- Submits to `?/updateRoles` form action
- Uses existing `api.updateMemberRoles(did, roles)` client method

No modal — roles are a simple 4-checkbox operation.

### Pre-existing Issue: Urgency Label Mismatch

The commerce needs page shows 'critical' in the UI filter but the DB constraint uses 'urgent'. Investigate and fix as part of V8.11c (before building the edit modal that would surface this mismatch).
